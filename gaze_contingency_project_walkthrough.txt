Gaze Contingency Project Walkthrough

What this project is
This is a Unity 6 mixed reality research experiment for the HTC Vive Focus Vision. A participant sees a passthrough view of a real table and a virtual bookshelf anchored to that table. On each round, they must find one target object among 56 spawned objects using eye-gaze dwell.

Presentation order in person
1. Show SampleScene and explain that most systems auto-attach at runtime.
2. Enter Play mode and show the first tap that starts the task.
3. Walk the code in this order:
- FindObjectGameManager.cs
- ChallengeSet.cs
- ShelfSpawner.cs
- ShapeObjectFactory.cs
- SpawnableObjectInfo.cs
- GazeHighlightManager.cs
- GazeCoverageTracker.cs
- HintGenerator.cs
- VoiceAssistantController.cs
- VoiceSynthesizer.cs
- GazeDataLogger.cs
- TrialDataLogger.cs
- SessionConfig.cs
- FindObjectUI.cs

Runtime architecture
Real table / Vive plane detection -> VivePlaneProvider + VivePlaneData -> FindObjectGameManager
FindObjectGameManager -> ChallengeSet for deterministic rounds
FindObjectGameManager -> ShelfSpawner for bookshelf layout and spawn points
FindObjectGameManager -> ShapeObjectFactory for shape/color object construction
User gaze -> XRGazeInteractor -> GazeHighlightManager for dwell and capture
User gaze history -> GazeCoverageTracker for fixation and search behavior
Round state + gaze state -> HintGenerator for aware vs unaware hints
HintGenerator -> VoiceAssistantController -> VoiceSynthesizer -> spoken feedback
All runtime state -> GazeDataLogger and TrialDataLogger -> CSV and JSON output
SessionConfig organizes participant/run folders

File breakdown

FindObjectGameManager.cs
This is the experiment brain. It owns the game state machine, the current round, current target, session start, transitions, spawning, and completion.
Key snippet:
[RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.AfterSceneLoad)]
static void AutoAttach()
{
    var spawner = FindObjectOfType<ObjectSpawner>();
    if (spawner != null && spawner.GetComponent<FindObjectGameManager>() == null)
        spawner.gameObject.AddComponent<FindObjectGameManager>();
    DisableTemplateSystems();
}
Meaning: the scene is intentionally minimal because the experiment logic attaches itself at runtime.

GazeHighlightManager.cs
This is the dwell-based selector.
Key snippet:
const float k_DwellDuration = 1.6f;
if (hoveredObj != null && hoveredObj == m_DwellTarget && !m_CapturedThisTarget)
{
    m_DwellTime += Time.deltaTime;
    if (m_DwellTime >= k_DwellDuration)
        CaptureObject(hoveredObj);
}
Meaning: the participant must maintain gaze on the same object long enough to capture it.

GazeCoverageTracker.cs
This summarizes recent fixation behavior.
Key snippet:

Important rule examples:
if (avgDuration < 0.6f && zoneSwitches > 4) return GazeBehavior.Erratic;
if (dominantZonePct > 0.7f && searchTime > 15f) return GazeBehavior.Stuck;
if (avgDuration > 1.5f && uniqueRatio > 0.6f) return GazeBehavior.Systematic;
Meaning: it turns raw gaze behavior into interpretable search-state labels.

string EvaluateGazeUnaware(float now)
{
    if (m_WrongCaptureTime > 0f && (now - m_WrongCaptureTime) < k_WrongCaptureWindow)
        return PickUnaware(k_GU_WrongCapture);
    return PickUnaware(k_GU_General);
}
Meaning: unaware mode gives generic encouragement, not gaze-sensitive support.

How to explain the aware vs unaware modes
Gaze-unaware mode: the assistant gives generic support and pacing. It does not react to what the participant is actually looking at.
Gaze-aware mode: the assistant uses gaze evidence and proximity logic to say whether the participant is on track, very close, or off target.

Data and state diagram
User taps detected table
  -> FindObjectGameManager.StartGame
  -> ShelfSpawner builds shelf layout
  -> ChallengeSet provides deterministic round target + distractors
  -> ShapeObjectFactory configures spawned objects
  -> User searches with gaze
  -> GazeHighlightManager accumulates dwell and captures object
  -> GazeCoverageTracker summarizes behavior
  -> HintGenerator chooses hint style depending on mode
  -> VoiceAssistantController sends text to VoiceSynthesizer
  -> VoiceSynthesizer plays spoken feedback
  -> GazeDataLogger writes per-frame CSV
  -> TrialDataLogger writes events and summary JSON
  -> SessionConfig organizes participant/run folders
